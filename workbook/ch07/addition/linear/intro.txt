
Tensor Nets

The program runs a repeating demo on the Pico 2 with Display Pack 2.0 (320x240).
It cycles through five states automatically, pausing 2 seconds between each.


Top area (y=0..30):
  A header line reading "TENSOR PRODUCT (A [cross] B)" followed by
  "Parallel Sessions:" to label what is below.

Middle area (y=60..110):
  Two vendor boxes side by side, each 80x50 pixels.
    VendorA (red tint, left):   x=20
    VendorB (yellow tint, right): x=140
  Each box shows: vendor name, current state, item name, and the session
  type string "!Coin->Apple" / "!Coin->Banana" (reads as: unlimited coin
  input, lollipop arrow, item output).
  Between them, a cross block character acts as the tensor symbol.
  When both vendors finish, "DONE" appears centered between them.

Lower area (y=140 downward):
  A proof net tree drawn with anti-aliased circles and lines.

    -o          (lollipop, root, at screen centre x=160 y=140)
    /  \
   x    x       (two tensor nodes at y=180, x=100 and x=220)
  / \  / \
 A   B B   A    (atom leaves at y=220)

  The "x" nodes represent tensor (⊗) and "-o" is the lollipop (⊸).
  Red dashed lines connect the dual atom pairs: A (left) <-> A (right)
  and B (left) <-> B (right). These are the axiom links that make the
  proof of (A⊗B) ⊸ (B⊗A) valid — A goes to A and B goes to B, just
  swapped sides.


THE FIVE DEMO STATES
--------------------
State 0 - startup:
  Both vendors are in IDLE, then AWAIT state.
  Two coin resources are created (MULTIPLICITY_UNLIMITED, uses_remaining=-1)
  and handed to VendorA and VendorB simultaneously. Both move to COIN_RECEIVED.

State 1 - VendorA delivers:
  vendor_deliver_item(0) consumes VendorA's coin (uses_remaining goes 0,
  consumed=true) and creates an Apple resource (MULTIPLICITY_LINEAR).
  VendorA moves to DELIVERING_ITEM.

State 2 - VendorB delivers:
  Same for VendorB: coin consumed, Banana resource created.
  VendorB moves to DELIVERING_ITEM.

State 3 - tensor completes:
  Both vendors are marked COMPLETE.
  update_tensor_session() checks that both are COMPLETE and sets
  tensor_sessions[0].both_complete = true, which triggers the "DONE" label.

State 4 - proof net highlight:
  main_proof_net->left (the left tensor node) gets highlighted=true,
  which makes it draw in white instead of its normal green.

After state 4 the demo resets: all counts go to zero and init_demo() runs again.


DATA STRUCTURES
---------------
vendor_t:
  Holds name, item name, current state (enum 0-5), coin resource index,
  item resource index, screen x/y, and color. Two instances live in the
  vendors[] array.

resource_t:
  Tracks a named resource with a multiplicity (LINEAR/AFFINE/UNLIMITED),
  uses_remaining (-1 = infinite), consumed flag, and owner vendor index.
  Resources accumulate in resources[] and are never freed mid-run
  (resource_count is just zeroed on reset).

tensor_session_t:
  Holds the two vendor indices and a both_complete flag.
  There is only one session (index 0) per run.

proof_node_t:
  A node in the proof net graph. Has a type (FORMULA, TENSOR, LOLLIPOP,
  etc.), a label string, screen x/y set by layout_proof_tree(), left/right
  child pointers, a dual pointer for axiom links, highlighted flag, and color.
  The tree built by build_tensor_proof_net() has this shape:

    lollipop ("-o")
    |- left:  tensor1 ("x")  (A left, B right)
    |- right: tensor2 ("x")  (B left, A right)

  Axiom links (dual pointers): a1<->a2 and b1<->b2.
  These are drawn as red dashed lines between the leaf nodes.


RENDERING PIPELINE
------------------
Every frame (~60 fps):
  1. fb_clear() fills the 320x240 framebuffer with black.
  2. draw_header() writes the title text.
  3. draw_vendor() draws each vendor box and its text into the framebuffer.
  4. draw_tensor_indicator() draws the tensor symbol and "DONE" if complete.
  5. draw_proof_net_node() recursively draws circles, edges, and labels
     for the proof net tree. For any node that has a dual pointer, a red
     dashed axiom link is drawn (each node draws it only if its own pointer
     address is lower than its dual's, to avoid drawing the line twice).
  6. display_blit_full() sends the entire framebuffer to the ST7789V2
     display controller via SPI+DMA.

The demo state machine (update_demo) advances at most once per 2000ms,
independent of the render loop.


REMAINING LIMITATIONS
---------------------
- The font table covers ASCII 32-122 only, so any Unicode symbols added
  to node labels in the future will render as blank spaces.
- The label centering formula (strlen * 4) works for single-char labels
  but drifts slightly for two-char labels like "-o".
- NODE_CUT is defined in the node type enum but no longer used; the axiom
  links are now represented directly via dual pointers on NODE_FORMULA nodes.
